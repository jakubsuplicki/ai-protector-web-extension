# Step 3 вҖ” Generate RBAC

**Time: 10 minutes**
**Input: tool registry from Step 2**
**Output: `rbac.yaml` with roles, inheritance, and default-deny**

---

## Why this step matters

RBAC is the single most impactful security control for agents.
Without it, every user can call every tool. With it, a customer
can search the FAQ but can't issue refunds or access admin secrets.

You don't write RBAC from scratch. AI Protector generates it from
your tool registry and risk profile, then you review and adjust.

---

## How RBAC generation works

AI Protector looks at your tools and applies these rules:

| Tool property | RBAC effect |
|--------------|-------------|
| `sensitivity: low` | Available to all roles |
| `sensitivity: medium` | Available to `support` and `admin` |
| `sensitivity: high` | Available to `admin` only |
| `sensitivity: critical` | Available to `admin` + `requires_confirmation` |
| `access: write` | `requires_confirmation: true` for non-admin |
| `returns_pii: true` | Post-tool PII redaction for `customer` role |

### Default role hierarchy

AI Protector generates 3 roles by default (you can rename or add more):

```
customer   вҶ’   lowest privileges, read-only, external users
    вҶ‘
support    вҶ’   inherits customer + medium-sensitivity tools
    вҶ‘
admin      вҶ’   inherits support + high/critical + write tools
```

Inheritance means: `admin` automatically gets everything `support` has,
and `support` gets everything `customer` has. You only define what's
**new** at each level.

## Generated output: rbac.yaml

Based on the Customer Support Copilot tool registry from Step 2:

```yaml
# rbac.yaml вҖ” generated by AI Protector
# Agent: Customer Support Copilot
# Generated: 2026-03-09
#
# Rule: default-deny. If (role, tool) is not listed вҶ’ BLOCKED.
# Inheritance: child role inherits all tools from parent.

roles:
  customer:
    description: "End-user customer вҖ” read-only access to public tools"
    tools:
      searchKnowledgeBase:
        scopes: [read]
        sensitivity: low
      getOrderStatus:
        scopes: [read]
        sensitivity: low

  support:
    description: "Support agent вҖ” customer tools + profile access"
    inherits: customer
    tools:
      getCustomerProfile:
        scopes: [read]
        sensitivity: medium
        # Note: returns PII вҖ” post-tool gate will redact for non-support roles

  admin:
    description: "System administrator вҖ” full access, write actions with confirmation"
    inherits: support
    tools:
      getInternalSecrets:
        scopes: [read]
        sensitivity: critical
        requires_confirmation: true
      issueRefund:
        scopes: [write]
        sensitivity: high
        requires_confirmation: true
```

### What this means in practice

| Action | customer | support | admin |
|--------|----------|---------|-------|
| Search FAQ | вң… | вң… | вң… |
| Check order status | вң… | вң… | вң… |
| View customer profile | вқҢ BLOCKED | вң… | вң… |
| Issue refund | вқҢ BLOCKED | вқҢ BLOCKED | вң… (confirm) |
| Access secrets | вқҢ BLOCKED | вқҢ BLOCKED | вң… (confirm) |

## Customization

The generated RBAC is a starting point. Common adjustments:

### Add a new role

```yaml
roles:
  supervisor:
    description: "Team lead вҖ” can issue refunds without confirmation"
    inherits: support
    tools:
      issueRefund:
        scopes: [write]
        sensitivity: high
        requires_confirmation: false  # supervisors skip confirmation
```

### Add scope restrictions

```yaml
  support:
    tools:
      getCustomerProfile:
        scopes: [read]          # can read but NOT update
        sensitivity: medium
        conditions:
          - own_customers_only   # optional: scope to assigned customers
```

### Remove confirmation for trusted roles

```yaml
  admin:
    tools:
      issueRefund:
        requires_confirmation: false  # admins skip confirmation
```

### Add rate limits per role

```yaml
  customer:
    tools:
      searchKnowledgeBase:
        scopes: [read]
        sensitivity: low
        rate_limit: 10           # max 10 calls per session
```

## Review checklist

Before moving on, verify:

- [ ] **Default-deny works** вҖ” roles not listed for a tool can't use it
- [ ] **Write tools require confirmation** вҖ” at least for customer/support
- [ ] **Critical tools are admin-only** вҖ” or at minimum require confirmation
- [ ] **Inheritance makes sense** вҖ” each child role only adds, never removes
- [ ] **PII tools have post-scanning** вҖ” `returns_pii: true` in tools.yaml
- [ ] **Rate limits are set** вҖ” especially for customer-facing tools
- [ ] **No role has access to ALL tools** вҖ” least privilege principle
